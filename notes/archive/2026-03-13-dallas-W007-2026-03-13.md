## What I Learned

1. **Bebop feature module pattern is strict and well-documented** — CONTRIBUTING.md has exhaustive rules for feature anatomy, atom conventions, and dependency flow. Following it exactly makes PRs clean.

2. **CSS Modules in Bebop use `@layer component`** (not `@layer components` as I initially expected). The existing patterns show `@layer application` for layout-level styles and `@layer component` for individual component styles.

3. **Jotai atom dispatch pattern** — The existing `chatModeAtoms.ts` uses action handler records mapping action types to state transforms. My progression atoms follow a switch-case variant that's equally valid but I could refactor to match the record pattern for consistency.

4. **office-bohemia worktree setup requires `yarn install`** — Unlike smaller repos, the monorepo needs a full yarn install in each worktree (takes ~13 minutes). The `.yarn/install-state.gz` in the main tree doesn't transfer.

5. **Test infrastructure requires `yarn build --to bebop`** first — Jest config depends on `@fluidx/office-bohemia-build-tools` which must be built before tests can run.

6. **Phase 1 vs Phase 2 separation is clean** — The atom layer is the same for both phases. Only `useProgressionStream` needs to change when Sydney adds structured NDJSON progression events. This was a good design decision by Rebecca.

## Patterns Established

- `features/progression/` module following exact design doc structure
