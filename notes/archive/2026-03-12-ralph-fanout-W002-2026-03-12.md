### Agent Architecture Pattern
Every agent follows this pattern:
```
agents/<name>/
  src/
    registry.ts              # Agent config: ID, model, versions, skills, flights, filters
    <agent-name>.ts          # Main agent class (WebSocket routing, orchestration)
    types.ts                 # Core types
    constants.ts             # Flight keys, constants
    hooks/                   # Agent-specific hooks (implements HookProvider)
    services/                # File watcher, messenger, logger, perf, state
    prompts/
      <version>/
        system-prompt.md     # EJS template - main system prompt
        rai.md               # Responsible AI guidelines
        agent-capabilities/  # Per-type descriptions
        intent-detection/    # Request interceptors
        skills/              # Skill definitions (SKILL.md with YAML frontmatter)
        subagents/           # Sub-agent definitions (.md with frontmatter)
        scripts/             # Runtime scripts
        overwrites/          # Flight-conditional replacements
```

### Agent Initialization Flow
```
1. Agent constructor → super(agentRegistry)
2. Hook registry init → HookProvider instances (Word, Excel, PPT)
3. initialize(context) → resolveConfig(context) applies flights/filters
4. ConfigResolver.resolve(registry, flights, context)
5. SystemPromptBuilder.buildPrompt(config, context, promptContext)  # EJS templates
6. setupWorkspace(config) → copies skills to /agent/.claude/skills/
7. Agent ready → run(query) → claude.query()
```

### Flight System
- Key format: `OfficeAgent.<AgentName>.<FeatureName>` (e.g., `OfficeAgent.EnableCreateAgent@1.1.0`)
- Enable via `.env`: `OAGENT_FLIGHTS=<flightKey>:true`
- Flights can override: version, model, subagents, skills, prompt files
- `applicableVersions[]` scopes flights to specific versions

### Grounding Pipeline (`@officeagent/grounding`)
4-stage enterprise data retrieval:
1. **Preprocessing** → query analysis
2. **Retrieval** → SearchOrchestrator / FetchOrchestrator
3. **Postprocessing** → result filtering
4. **Storage** → context injection

Supports: Files, Emails, Meetings, Chats, Transcripts, People

### Docker Runtime
- **Base:** `mcr.microsoft.com/officepy/codeexecutionjupyter4` (multi-stage build)
- **Includes:** Node.js 24, Python (OfficePy), LibreOffice 26.2, Poppler 26.02, Azure CLI, .NET
- **Key env:** `CLAUDE_CODE_MAX_OUTPUT_TOKENS=64000`
- **Health:** `curl http://localhost:6010/health`
- **Scaling:** `docker-compose --scale officeagent=3`

### Eval Framework (`eval/`)
- Python pytest-based, integrated via `oagent eval` CLI
- Aliases: `ppt.groundingeval`, `word.groundingeval`, `excel.pairwise`, etc.
- Output: `eval/outputs/<run_name>/eval_report_*.json`
- Usage: `oagent eval ppt.groundingeval -n 10`

### Skills vs Sub-Agents
- **Skills:** Instructions + scripts the main agent follows (general capabilities)
- **Sub-agents:** Independent Claude instances with isolation (pipeline steps)
