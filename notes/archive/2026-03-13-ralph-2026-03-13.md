## What I Learned

1. **All 3 office-bohemia PRs are still active with no reviewer votes** — PR-4959092 (cowork route), PR-4959405 (W003 design doc), and PR-4964594 (W007 progression UI) all have `status: "active"`, `mergeStatus: "succeeded"` (no conflicts), and only the "Bebop PR Reviewers" group assigned with vote 0 (no vote).

2. **ADO `lastMergeCommit` is a test merge, not actual merge** — The merge commit shown in ADO PR responses (with comment "Merge pull request NNNN...") is ADO's automatic test merge to check mergeability. The actual PR status field (`status: active/abandoned/completed`) is the source of truth.

3. **No individual human reviewers assigned** — All 3 PRs only have the "Bebop PR Reviewers" team group as reviewer. No individual votes have been cast.

4. **Fallback when ADO MCP tools unavailable** — Use `azureauth aad --resource 499b84ac-1321-427f-aa17-267ca6975798 --client 872cd9fa-d31f-45e0-9eab-6e460a02d1f1 --tenant 72f988bf-86f1-41af-91ab-2d7cd011db47 --output token --mode broker` to get a bearer token, then call ADO REST API directly via curl.

## Patterns

- PR sync: fetch via ADO API, map status/mergeStatus/reviewer votes, compare with local tracker.
- When all reviewer votes are 0 and there's at least one reviewer → `reviewStatus: "pending"`.
- `mergeStatus: "succeeded"` → `buildStatus: "passing"` (merge check succeeded).
- `--mode broker` works for azureauth; `--mode iwa` fails.
