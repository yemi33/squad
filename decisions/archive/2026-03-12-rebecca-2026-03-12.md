## Architecture

### High-Level Summary

bebop-desktop is a **single-page React application** built with Vite, serving as a **designer's UX prototype** for Microsoft Copilot ("Bebop") interaction patterns. It is NOT a production app — it's a prototype/demo tool for exploring AI chat UX, artifact creation, workspace management, and agent interaction flows.

### Tech Stack
- **Build:** Vite 6 + TypeScript 5 + React 18
- **UI Framework:** Fluent UI React v9 + styled-components
- **State:** Zustand (multiple stores)
- **Database:** Dexie (IndexedDB) for persistence
- **Animation:** Motion (framer-motion successor)
- **Rich Text:** Lexical editor
- **Canvas/Board:** @xyflow/react (React Flow)
- **Multiplayer:** Fluid Framework (Azure Fluid Relay)
- **Auth:** @m365-core-design/sandbox (Microsoft internal sandbox auth)
- **AI Calls:** Sandbox `ChatCompletions` API (not direct OpenAI)
- **Path alias:** `@/` → `./src/`

### State Architecture (Zustand)

The app uses **8 Zustand stores** (no single global store):

| Store | File | Purpose |
|-------|------|---------|
| `useChatStore` | `useChatStore.ts` | Chat sessions, messages, AI completions, streaming |
| `useUiStore` | `uiStore.ts` | UI state: pages, menus, editor focus, attachments, scroll |
| `useSettingsStore` | `settingsStore.tsx` | User prefs (persisted to localStorage via zustand/persist) |
| `useArtifactStore` | `artifactStore.ts` | Artifact CRUD, backed by IndexedDB |
| `useFlowPlayerStore` | `flowPlayerStore.ts` | JTBD flow playback engine |
| `useWorkspaceStore` | `useWorkspaceStore.ts` | Workspace grouping of chats/artifacts |
| `useNodeStore` | `useNodeStore.ts` | Board/canvas node management |
| `useMultiplayerStore` | `useMultiplayerStore.ts` | Fluid Framework multiplayer state |
| `useCanvasStore` | `canvasStore.ts` | Canvas selection/editing state |
| `useMeetingsStore` | `useMeetingsStore.ts` | Meeting data from MS Graph |
| `useFilesStore` | `useFilesStore.ts` | File data |
| `liveActivityStore` | `liveActivityStore.ts` | Live activity tracking |
| `themeStore` | `themeStore.ts` | GenUX theming |

**Key pattern:** Stores directly import and call each other (e.g., `flowPlayerStore` calls `useChatStore.getState()`, `artifactStore` calls `useWorkspaceStore.getState()`). This creates tight coupling between stores.

### Data Layer

- **Dexie (IndexedDB):** `ChatDatabase` class manages 3 tables:
  - `chatSessions` — user chat history
  - `artifacts` — generated artifacts (documents, images, tables, presentations)
  - `workspaces` — groupings of chats/artifacts
- **Default data:** Hardcoded in `src/data/` (defaultChatSessions, defaultChatArtifacts) — loaded at runtime, NOT stored in DB
- **Flow data:** Hardcoded JTBD flow definitions in `src/data/flows/`

### Flow Player System (Critical Architecture)

The **Flow Player** is the prototype's demo engine — it replays scripted JTBD (Jobs To Be Done) scenarios:

1. `userJobs.ts` defines 22 JTBD scenarios (enum + descriptions)
2. `src/data/flows/` contains per-JTBD flow definitions (each has `flow.tsx`, `artifacts.tsx`, `inline.tsx`)
3. `getFlow.ts` maps `globalRenderIndex` → `flowId`
4. `flowPlayerStore.ts` executes flows step-by-step:
   - Adds messages to chat store
   - Executes CoT processing steps with delays
   - Supports streaming simulation
   - Manages artifact creation via flow
5. **JobNavigator** (keyboard-driven: backtick to toggle, arrow keys to navigate)

Data flow: `globalRenderIndex` → `getFlowIdFromRenderIndex()` → `setFlow()` → `executeStep()` → `chatStore.addMessage()`

### Tool System

Chat completions support tool calling with 6 tools:
- `enterpriseSearch` — M365 Graph search
- `generateImage` — Image generation
- `getInstructions` — Format instructions lookup
- `createArtifact` — Create document/presentation/table
- `updateArtifact` — Update existing artifact
- `createTheme` — Create GenUX theme

### GenUX System

`src/lib/gen-ux/` — A custom markdown-to-component rendering system:
- `componentRegistry.tsx` — Maps markdown elements to Fluent UI components
- `slideComponentRegistry.tsx` — Presentation slide rendering
- `markdownIdInjection.ts` — Injects targeting IDs into markdown for editing
- `markdownTargeting.ts` — Block-level selection and targeting
- `markdownEditing.ts` — Slide reordering
- Theming system at `lib/gen-ux/theming/`

### Cross-Platform

Three rendering modes (`PreviewFormat` enum):
- **Desktop** — full Prototype layout
- **Mobile** — MobilePrototype with optional DeviceFrame
- **SideBySide** — dual preview

Mobile has its own component tree under `src/components/Mobile/iOS/`.

### Event Bus

Uses `document.dispatchEvent/addEventListener` for cross-store communication:
- `exit-flow` — cleanup flow player artifacts
- `open-artifact` / `close-artifact` — artifact overlay
- `artifact-update` — broadcast artifact changes

## Patterns

1. **Zustand stores with auto-generated setters:** Both `uiStore` and `settingsStore` use a `generateSetters()` helper that creates `setX` actions from state keys. Clever but makes types verbose.

2. **Store cross-calls:** Stores freely call `otherStore.getState()` to coordinate. No middleware or event-driven decoupling.

3. **Dexie for persistence, Zustand for state:** Artifacts are loaded from DB into store on init, then store is source of truth. DB writes are fire-and-forget.

4. **Flow data = hardcoded TSX:** Each JTBD flow is a hand-authored TypeScript file defining messages, CoT steps, artifacts, and delays. This is the core "prototype" nature — it's scripted demos, not live AI.

5. **styled-components + Fluent tokens:** Most components use `styled-components` with Fluent UI `tokens` for theming.

6. **No router:** Page navigation is handled via `Page` enum in `uiStore.currentPage`. No React Router.

7. **MARK comments:** Code uses `// MARK:` comments (Xcode convention) for code organization.

8. **Iframe messaging:** `IframeMessageInterface` syncs state between desktop and mobile previews via postMessage.

## Gaps

1. **No CLAUDE.md:** The repo has no `CLAUDE.md` for agent guidance. Only `README.md` with basic setup instructions and `memory-bank/` with previous context from another AI tool.

2. **Circular store dependencies:** `ChatDatabase` imports and calls 6+ stores in its `reset()` method. Stores also freely call each other. This creates fragile initialization order and makes testing/mocking difficult.

3. **No tests:** Zero test files found. The prototype nature makes this expected but still a gap for any production-bound code.

4. **No routing library:** Single-page with enum-based navigation. Fine for prototype, but would need a router for deep linking or URL-based state.

5. **`streaming` field typed as `any`:** `FlowStep.streaming` is `any` — should have a proper type.

6. **Dead code:** Several commented-out imports and features (RightRail, useFlowDeepLink, various localStorage clears).

7. **Flow data coupling:** The `getFlowIdFromRenderIndex` function uses a hardcoded switch statement mapping indices to flows. Adding a new flow requires updating multiple files.

8. **No error boundaries:** No React error boundaries found — a crash in any component could take down the whole app.

## Recommendations

1. **Add a CLAUDE.md** at repo root with:
   - Project description and purpose (prototype, not production)
   - Build/run instructions (`npm run setup && npm install && npm start`)
   - Architecture overview (stores, flows, GenUX)
   - Key conventions (styled-components, MARK comments, Zustand patterns)
   - Path alias (`@/` = `./src/`)

2. **For any agent working on this repo:** Understand that this is a **demo prototype** — most "AI" behavior is scripted via Flow data files. The actual AI integration goes through the `@m365-core-design/sandbox` `ChatCompletions` API, not direct OpenAI calls.

3. **When adding new JTBD flows:** Follow the existing pattern:
   - Create `src/data/flows/<StepNN-JN>/` with `flow.tsx`, `artifacts.tsx`, `inline.tsx`
   - Register in `getFlow.ts` (both `allFlows` array and `getFlowIdFromRenderIndex` switch)
   - Update `userJobs.ts` if adding a new job

4. **When modifying stores:** Be aware of cross-store dependencies. Test that initialization order still works. The stores are tightly coupled.

5. **GenUX rendering:** The `componentRegistry.tsx` is the central mapping of markdown → React components. To add new artifact rendering capabilities, extend this registry.

## Key Learnings for Team

1. **Not a monorepo.** Single `package.json`, single Vite app. No workspaces, no turborepo.
2. **ADO org is `open-studio`, project is `Prototypes`.** Different from the other repos (office/ISS, office/OC).
3. **No `node_modules` present.** Need to run `npm run setup && npm install` first.
4. **Uses private npm registry** via `.npmrc` for `@m365-core-design/sandbox` package.
5. **The Flow Player is the heart of the prototype.** Most "impressive" demos are scripted flows, not live AI.
6. **Zustand stores are the architectural backbone.** ~13 stores, heavily interconnected.
7. **GenUX = custom markdown rendering.** Documents, presentations, and tables are rendered from markdown via a component registry.
