# Lambert Learnings — 2026-03-12

## PR-4959180: Add /cowork route to Bebop webapp

### What I Learned

1. **Bebop thin-route pattern is highly uniform**: Route files under `apps/bebop/src/routes/` follow a strict template — `head()` for meta, `loader()` for data, `component` for rendering. When a new route is just a passthrough to an existing component (like `Chat`), the file is essentially a copy of the index route with different metadata. This makes review straightforward: diff against the canonical index route.

2. **routeTree.gen.ts has 8+ registration points**: Adding a route requires touching the import, `.update()` call, 3 type interfaces (`FileRoutesByFullPath`, `FileRoutesByTo`, `FileRoutesById`), 3 union types inside `FileRouteTypes`, the `@tanstack/react-router` module declaration, and the parent's `Children` interface + object. Missing any one of these will cause TypeScript compilation errors.

3. **resetToken is intentionally non-persistent**: The `crypto.randomUUID()` in loaders is by design — it forces component remount and state reset on every navigation. This is not a bug or performance issue.

4. **Feature gating for thin routes**: Per Loop's feature maturity policy, thin passthrough routes that render existing gated components don't need their own feature gate. The gate should be added when the route develops its own distinct UI.

### Patterns for Future Reviews

- Cross-reference new routes against `_mainLayout.index.tsx` as the canonical pattern.
- Check all 8+ registration points in `routeTree.gen.ts` — it's the most error-prone part of adding a route.
- Comments explaining non-obvious patterns (like resetToken) should be propagated to new route files for maintainability.
