# Lambert Learnings — 2026-03-13

## PR Status Sync Task

### Patterns Discovered

1. **ADO PR status codes**: `status: 1` = active, `status: 2` = abandoned, `status: 3` = completed/merged. The `mergeStatus` field is separate — `mergeStatus: 3` means the trial merge succeeded (no conflicts), NOT that the PR is merged.

2. **ADO reviewer vote mapping**: vote values — `10` = approved, `5` = approved-with-suggestions, `0` = no vote, `-5` = waiting for author, `-10` = rejected. Group reviewers (like "Bebop PR Reviewers") often show vote=0 even when individual members may have reviewed — the group vote stays 0 unless explicitly set.

3. **Local tracker drift**: Both PRs had `reviewStatus: "approved"` in the local tracker but ADO showed only a group reviewer with vote=0. This suggests the "approved" status was set based on squad agent reviews (Ripley), not ADO's native review system. The sync correctly updates to reflect ADO's ground truth.

### Gotchas

- **Group reviewers mask individual votes**: When a reviewer is a group (like "Bebop PR Reviewers"), the group's vote field may be 0 even if individual members within the group have voted. The `repo_get_pull_request_by_id` API only returns the top-level reviewer entries, not nested member votes.

- **Merge commit in active PRs**: Both active PRs had a `lastMergeCommit` with a "Merge pull request..." comment. This is ADO's trial merge (to check for conflicts), not an actual completion merge. Don't confuse this with the PR being merged — always check `status` field.

### Conventions

- Always use the project's `repositoryId` (not repo name) when calling `repo_get_pull_request_by_id`
- Map `mergeStatus: 3` → `buildStatus: "passing"`, `mergeStatus: 2` → `buildStatus: "conflicts"`
- Added `buildStatus` field to tracker entries — this was missing before and is useful for dashboard visibility
